Basic of Asynchronous Device I/O

The two types I/O are illustrated in the fllowing figure.

I/O bound:asynchronous I/O is generally a good way to optimize processing efficiency.
CPU bound:the overhead of processing kernel I/O requests and kernel signals may make asynchronous I/O less beneficial.
Device I/O: is one of the slowest and most unpredictable(不可预估) compared to other operations carried out by a computer.

Device driver: which assumes(承担) the responsibility of actually performing the I/O.

Asynchronous I/O: When a thread issues an I/O request to a device, this I/O request is passed to a device driver.While the device driver waits for device to respond,the thread is not suspended instead continue executing other tasks.Afer proccessing the I/O request driver must notify the thread.

Queue asynchronous I/O request

Queuing asynchronous I/O request is the enssence of desining a high-performance application.

To access a device asynchronously

  1. must open device specifying the FILE_FLAG_OVERLAPPEDflag in the dwFlagAndAttributes parameter.

  2. must pass the address to an initilized OVERLAPPED structure via the pOverlapped paramer.

The OVERLAPPED Structure

Offset and OffsetHigh: Indicate 64-bit offset in the file where you want the I/O operation to begin.
Every file kenerl object has a file pointer,in synchornous I/O request,system access the file at the location identified by the file pointer.But in asynchornous I/O request,to avoid the confusion of multiple asynchorous calls to the same object.
the file pointer is ignore by the system and all asynchornous I/O request must specify the start file offset.

hEvent: Used by one of the four methods available for receiving I/O completion notifications.
when using the alertable I/O notification,this param can be used for own.

Caveats

  1. The device driver doesn’t have to process queued I/O requests in a FIFO,typically executes I/O requests out of order if doing so helps performance.

  2. The asynchornous I/O request sometimes process in synchornous.

  3. The data buffer and OVERLAPPED structure must not be moved or destroyed until the I/O compeleted.

Canceling Queued Device I/O Requests

  1. BOOL CancelIo(HANDLE hFile); cancel all I/O requests specified by thread handle.

  2. close the device handle; cancel all I/O requests regardless of which thread issued.

  3. when a thread dies,the system automatically cancels all I/O requests issued by the thread.

  4. BOOL CancelIoEx(HANDLE hFile,LPOVERLAPPED pOVerlapped);cancel a specify thread’s specify I/O request by HANDLE and OVERLAPPED address.

Receiving Completed I/O Request Notifications

Windows offers four different methods for receiving I/O completion notifications.

Signaling a device kernel object

The ReadFile and WriteFile functions set the device kernel object to the nosignaled state just before queuing the I/O request.When the device driver completes the request,the driver sets the the device kernel object to the signaled state.
thread can determine kernel object signaled by calling WaitForSingleObject or WaitForMultipleObjects.

Alertable I/O

Alertable I/O is horrible and should be avoid.But Microsoft added some infrastructure is useful and valuable.
APC queue: Whenever a thread is created,the system also creates a queue that is associated with the thread.This queue is called the asynchronous procedure call (APC) queue.

the detail of alertable I/O

When issue an asynchronous I/O request with * EX functions,the functions pass the address of completion routine to the device driver.
When the device driver has completed the I/O request,it appends an entry in the thread’s APC queue.The entry contains the address of the completion routine and the address of the OVERLAPPED structrue.
When the thread in alertable state,the system checks thread’s APC queue,if at least one entry in queue,the system does not put thread to sleep.
In alertable I/O completes,the device driver ignore OVERLAPPED structure’s hEvent member.

I/O completion port

An I/O completion port is the most complex kernel object.
call CreateIoCompletionPort to create an I/O completion port.

The function performs two different tasks:1.Create an I/O completion port.2.associates a device with an I/O completion port.
When you create an I/O completion port,the kernel actually create five different data structure.

  1. The first data structure is a device list indicating the device associated with the port.
  2. The second data structure is an I/O completion queue.
    When an asynchronous I/O request completes,the system checks to see whether the device is associated with a completion prot,if it is,the system appends an entry to the end of the I/O completion queue.

Basiclly,the job of GetQueuedCompletionStatus is to put the calling thread to sleep until an entry appears in I/O completion queue.

  1. The third data structure is an wait thread queue.
    Storage the ID of each thread in thread pool which calls GetQueueCompletionStatus.
    Enable the I/O completion port know which threads are currently waiting to handle completed I/O requests.
  2. The fourth data structure is a released thread list.
    When a completion port wakes a thread,the completion port places the thread’s ID in the released thread list.This allows the completion port to remember which threads it awakened.
  3. The fifth data structure is paused thread list
    If a released thread calls any function that places the thread in a wait state,the completion port detects this and updates its internal data structure by moving the thread’s ID from released thread list to the paused thread list.

This function appends a completed I/O notifiction to an I/O completion port’s queue.
